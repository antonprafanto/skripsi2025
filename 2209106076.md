## B. Hal-Hal yang Perlu Diperbaiki

### Bagian 1: Masalah Isi/Substansi

#### 1.1 Dataset Terlalu Sedikit

**Masalahnya:**
> Kamu menulis akan mengumpulkan data selama 2 hari dengan interval 3 menit. Ini menghasilkan sekitar 960 data saja.

**Kenapa ini masalah?**
- Machine learning butuh data yang banyak untuk "belajar" dengan baik
- 960 data terlalu sedikit, model bisa jadi tidak akurat
- Risiko overfitting (model hanya hafal data training, tidak bisa generalisasi)

**Solusinya:**
- Kumpulkan data minimal **2000-3000 sampel**
- Perpanjang waktu pengumpulan menjadi **1-2 minggu**
- Atau perbanyak variasi kondisi (pagi, siang, malam, setelah hujan, setelah penyiraman, dll)

---

#### 1.2 Tidak Jelas Bagaimana Menentukan Label "Subur" atau "Tidak Subur"

**Masalahnya:**
> Kamu tidak menjelaskan bagaimana cara menentukan bahwa suatu data termasuk "Subur" atau "Tidak Subur" (ground truth)

**Kenapa ini masalah?**
- Supervised learning (seperti CART) butuh label yang benar sebagai acuan
- Kalau labelnya asal-asalan, model yang dihasilkan juga tidak bisa dipercaya

**Solusinya:**
- Jelaskan kriteria pelabelan berdasarkan kombinasi parameter (contoh: jika pH 5.5-7 DAN kelembapan 40-60% DAN suhu 22-28°C = Subur)
- Atau validasi dengan pakar pertanian/agronomi
- Atau bandingkan dengan hasil pengujian laboratorium tanah

---

#### 1.3 Tidak Jelas Di Mana Model CART Dijalankan

**Masalahnya:**
> Kamu tidak menjelaskan apakah algoritma CART dijalankan di:
> - ESP32 (embedded/di dalam mikrokontroler)?
> - Server cloud?
> - Aplikasi mobile?

**Kenapa ini masalah?**
- Ini mempengaruhi desain sistem secara keseluruhan
- ESP32 punya keterbatasan memori dan processing power
- Penguji pasti akan menanyakan hal ini

**Solusinya:**
Tambahkan **diagram arsitektur sistem** yang menjelaskan:
```
[Sensor] → [ESP32] → [WiFi] → [Server/Cloud] → [Mobile App]
                                    ↓
                           [Model CART dijalankan di sini]
```

Atau jika model di ESP32:
```
[Sensor] → [ESP32 + Model CART] → [WiFi] → [Server] → [Mobile App]
```

---

#### 1.4 Pohon Keputusan di Gambar 3.3 Bukan Hasil Training

**Masalahnya:**
> Pohon keputusan di Gambar 3.3 terlihat seperti aturan yang kamu buat sendiri (rule-based), bukan hasil training algoritma CART dari data

**Kenapa ini masalah?**
- CART seharusnya membangun pohon keputusan secara otomatis dari data
- Threshold (batas nilai) seharusnya ditentukan oleh algoritma berdasarkan Gini Index, bukan ditentukan manual

**Solusinya:**
- Jelaskan bahwa Gambar 3.3 adalah **"rancangan awal"** atau **"hipotesis"**
- Tambahkan keterangan bahwa pohon final akan dihasilkan dari proses training
- Atau tunjukkan contoh perhitungan Gini Index untuk menentukan split point

---

#### 1.5 Jenis Tanaman Hias Tidak Spesifik

**Masalahnya:**
> Kamu tidak menyebutkan jenis tanaman hias apa yang akan diteliti

**Kenapa ini masalah?**
- Setiap tanaman punya kebutuhan berbeda (Aglonema vs Kaktus vs Anggrek)
- Threshold yang optimal untuk satu tanaman belum tentu cocok untuk tanaman lain

**Solusinya:**
- Fokuskan penelitian pada **2-3 jenis tanaman hias** yang spesifik
- Contoh: "Penelitian ini fokus pada tanaman Aglonema dan Monstera"
- Tambahkan di Batasan Masalah

---

#### 1.6 Sensor pH Tidak Spesifik

**Masalahnya:**
> Kamu hanya menampilkan gambar sensor pH tanpa menyebutkan tipe/merek spesifiknya

**Kenapa ini masalah?**
- Sensor pH ada banyak jenisnya dengan akurasi berbeda-beda
- Penguji akan bertanya sensor apa yang digunakan

**Solusinya:**
- Sebutkan tipe sensor (contoh: "Sensor pH Tanah Analog SEN0161" atau "Soil pH Sensor Module")
- Tambahkan spesifikasi teknis (range pengukuran, akurasi, tegangan operasi)
- Jelaskan prosedur kalibrasi sensor

---

#### 1.7 Tidak Ada Perbandingan dengan Algoritma Lain

**Masalahnya:**
> Kamu memilih CART tapi tidak menjelaskan kenapa tidak memilih algoritma lain seperti Random Forest, Naive Bayes, atau SVM

**Kenapa ini masalah?**
- Penguji akan bertanya "kenapa CART? kenapa tidak pakai Random Forest yang biasanya lebih akurat?"

**Solusinya:**
Tambahkan justifikasi pemilihan CART, misalnya:
- CART menghasilkan model yang interpretable (mudah dipahami)
- CART cocok untuk data numerik kontinu
- CART ringan untuk dijalankan di embedded system
- Atau tambahkan perbandingan dengan 1-2 algoritma lain sebagai baseline

### Bagian 2: Masalah Metodologi

#### 2.1 Tidak Ada Strategi Menangani Imbalanced Data

**Masalahnya:**
> Kamu tidak menjelaskan bagaimana jika jumlah data "Subur" jauh lebih banyak dari "Tidak Subur" (atau sebaliknya)

**Kenapa ini masalah?**
- Jika data tidak seimbang, model cenderung bias ke kelas mayoritas
- Contoh: 90% data Subur, 10% Tidak Subur → model bisa selalu prediksi "Subur" dan dapat akurasi 90% tapi tidak berguna

**Solusinya:**
Tambahkan rencana handling imbalanced data:
- SMOTE (Synthetic Minority Over-sampling Technique)
- Undersampling kelas mayoritas
- Atau gunakan stratified sampling saat split data

---

#### 2.2 Tidak Ada Strategi Pruning

**Masalahnya:**
> CART rentan overfitting, tapi kamu tidak menjelaskan strategi pruning

**Kenapa ini masalah?**
- Tanpa pruning, pohon keputusan bisa terlalu kompleks
- Model jadi "hafal" data training tapi tidak bisa prediksi data baru dengan baik

**Solusinya:**
Tambahkan penjelasan tentang:
- **Pre-pruning**: batasi kedalaman pohon (max_depth), minimum sampel per leaf
- **Post-pruning**: cost-complexity pruning setelah pohon terbentuk
- Sebutkan parameter yang akan digunakan (contoh: max_depth=5, min_samples_leaf=10)

---

#### 2.3 Kenapa 70:30? Tidak Ada Justifikasi

**Masalahnya:**
> Kamu menggunakan split 70% training dan 30% testing tanpa penjelasan

**Kenapa ini masalah?**
- Dengan dataset kecil (< 1000), split 70:30 bisa menghasilkan hasil yang tidak stabil
- Penguji akan bertanya kenapa tidak pakai cross-validation

**Solusinya:**
- Jelaskan alasan pemilihan rasio 70:30
- Atau pertimbangkan menggunakan **k-fold cross validation** (k=5 atau k=10) untuk hasil yang lebih reliable
- Tambahkan penjelasan di bagian metodologi

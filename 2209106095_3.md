## Pertanyaan 1: Apakah Baseline MySQL Sudah Cukup Kuat?

### Jawaban: **Sudah cukup, tapi perlu sedikit penjelasan tambahan.**

Penetapan MySQL `ORDER BY`, `WHERE`, dan `LIKE` sebagai baseline **sudah tepat** untuk level skripsi S1. Kamu tidak perlu masuk terlalu dalam ke query plan atau indexing optimization.

**Yang perlu kamu lakukan:**

1. **Jelaskan secara sederhana** di BAB III bahwa baseline menggunakan query MySQL standar tanpa optimasi khusus (tanpa menambah index manual selain primary key).

2. **Tambahkan 1-2 kalimat** seperti ini di bagian batasan masalah atau metodologi:

   > *"Pengujian baseline MySQL menggunakan konfigurasi default tanpa optimasi index tambahan, sehingga hasil perbandingan mencerminkan kondisi umum penggunaan database pada sistem web sederhana."*

3. **Jangan masuk ke EXPLAIN atau query plan** jika kamu tidak yakin bisa menjelaskannya saat sidang. Cukup sampaikan bahwa kamu membandingkan "mekanisme standar MySQL" vs "algoritma di lapisan aplikasi".

**Kesimpulan:** Baseline-mu sudah cukup. Fokus saja pada perbandingan waktu eksekusi, tidak perlu analisis query plan yang terlalu teknis.

## Pertanyaan 2: Apakah Fokus pada Average Case Sudah Tepat?

### Jawaban: **Ya, sangat tepat.**

Pendekatan ini **sudah benar** dan justru menunjukkan pemahaman metodologi yang baik. Berikut alasannya:

1. **Average case merepresentasikan penggunaan nyata.** Dalam sistem praktikum, data tidak selalu terurut sempurna (best case) atau terbalik sempurna (worst case). Pengguna akan mencari dan mengurutkan data dalam kondisi acak.

2. **Best/worst case sebagai pendukung sudah tepat.** Ini menunjukkan kamu memahami karakteristik algoritma secara teoritis, tapi tetap realistis dengan fokus pada kondisi nyata.

**Saran penulisan di BAB III atau BAB IV:**

> *"Pengujian performa utama dilakukan pada skenario average case karena merepresentasikan kondisi penggunaan umum sistem web. Skenario best case dan worst case digunakan sebagai analisis pendukung untuk memahami batas kinerja algoritma pada kondisi ekstrem."*

**Kesimpulan:** Pertahankan pendekatan ini. Jangan ubah.

## Pertanyaan 3: Apakah Pembatasan pada Server Processing Time Sudah Aman?

### Jawaban: **Ya, ini justru pendekatan yang benar secara metodologi.**

Membatasi pengukuran pada **server processing time** tanpa network latency adalah keputusan yang **tepat** karena:

1. **Mengisolasi variabel.** Network latency sangat fluktuatif dan dipengaruhi banyak faktor di luar kendalimu (kecepatan internet, jarak server, dll). Dengan mengisolasi server processing time, kamu bisa membandingkan performa algoritma secara lebih adil.

2. **Pengujian di localhost.** Karena kamu menguji di localhost (seperti yang tertulis di metodologi), network latency praktis tidak ada. Jadi pengukuranmu sudah tepat.

**Yang perlu kamu tulis di Batasan Masalah:**

> *"Pengukuran waktu eksekusi difokuskan pada server processing time, yaitu waktu yang dibutuhkan server untuk memproses algoritma dan query database. Pengukuran tidak mencakup network latency atau waktu rendering di browser karena pengujian dilakukan pada lingkungan localhost."*

**Kesimpulan:** Ini bukan kelemahan, tapi justru **kekuatan metodologi**. Pertahankan.

## Pertanyaan 4: Apakah Penggunaan Data Sintetis Sudah Tepat?

### Jawaban: **Ya, sangat tepat dan umum dilakukan dalam penelitian.**

Penggunaan data sintetis untuk skenario best/worst case dan variasi ukuran dataset adalah **praktik standar** dalam penelitian algoritma. Berikut alasannya:

1. **Best case dan worst case sulit ditemukan secara alami.** Data riil jarang dalam kondisi terurut sempurna atau terbalik sempurna. Maka data sintetis diperlukan untuk menguji kondisi ekstrem ini.

2. **Variasi ukuran data untuk skalabilitas.** Data riilmu hanya 624 (2024) dan 255 (2025). Untuk menguji skalabilitas hingga 5000 entri, data sintetis adalah solusi yang valid.

**Yang perlu kamu tulis di BAB III atau BAB IV:**

> *"Data sintetis digunakan untuk pengujian skenario best case, worst case, dan variasi ukuran dataset karena kondisi tersebut tidak selalu tersedia secara alami pada data riil. Data sintetis dibangkitkan dengan karakteristik yang menyerupai data praktikan (nama, NIM, nilai) untuk memastikan relevansi pengujian."*

**Penegasan untuk BAB IV:**

Di BAB IV, jelaskan bahwa:
- Pengujian **sistem terintegrasi (end-to-end)** menggunakan **data riil** dari TPAI
- Pengujian **algoritma murni** (untuk melihat best/worst/average case) menggunakan **data sintetis**

**Kesimpulan:** Pendekatan ini sudah benar. Pastikan kamu menjelaskan alasannya dengan jelas.

## Pertanyaan 5: BAB IV Fokus ke Analisis Teknis atau Dampak Efisiensi?

### Jawaban: **Keduanya, tapi dengan porsi yang seimbang.**

Ini pertanyaan yang sangat penting. Rekomendasiku:

### Struktur BAB IV yang Disarankan:

**4.1 Hasil Implementasi Sistem**
- Screenshot sistem yang sudah jadi
- Penjelasan singkat fitur-fitur utama
- *Porsi: 15-20%*

**4.2 Hasil Pengujian Fungsional (Black Box)**
- Tabel hasil pengujian login, registrasi, fitur praktikan/asisten/PJ
- Kesimpulan: sistem berjalan sesuai spesifikasi
- *Porsi: 15-20%*

**4.3 Hasil Evaluasi Kinerja Algoritma**
- Tabel waktu eksekusi Quick Sort vs ORDER BY
- Tabel waktu eksekusi Binary Search vs WHERE/LIKE
- Grafik perbandingan berdasarkan ukuran data
- **Analisis sederhana:** mana yang lebih cepat, berapa selisihnya, apa pola yang terlihat
- *Porsi: 30-35%*

**4.4 Pembahasan dan Implikasi**
- Hubungkan hasil teknis dengan **manfaat praktis**
- Contoh: *"Dengan waktu pencarian X ms menggunakan Binary Search dibandingkan Y ms menggunakan WHERE, sistem dapat memproses pencarian data praktikan Z kali lebih cepat dibandingkan pencarian manual yang membutuhkan 10-60 detik (berdasarkan hasil wawancara)."*
- *Porsi: 20-25%*

### Tips Penting:

1. **Jangan terlalu teknis.** Kamu tidak perlu menjelaskan kompleksitas Big-O secara mendalam atau membuat analisis statistik yang rumit. Cukup tampilkan data, buat tabel/grafik, dan jelaskan hasilnya dengan bahasa sederhana.

2. **Selalu hubungkan dengan konteks.** Setelah menyajikan angka, selalu tambahkan kalimat seperti:
   - *"Hasil ini menunjukkan bahwa..."*
   - *"Dalam konteks sistem praktikum, hal ini berarti..."*
   - *"Dibandingkan dengan proses manual yang membutuhkan X menit, sistem ini..."*

3. **Gunakan data wawancara sebagai pembanding.** Kamu punya data bahwa pencarian manual butuh 10-60 detik. Ini bisa jadi baseline non-teknis yang bagus.

**Contoh kalimat pembahasan yang bagus:**

> *"Hasil pengujian menunjukkan bahwa Binary Search membutuhkan rata-rata 0.5 ms untuk menemukan data praktikan dari 1000 entri, sedangkan query WHERE MySQL membutuhkan 1.2 ms. Meskipun selisihnya terlihat kecil dalam satuan milidetik, dalam konteks penggunaan sistem oleh asisten yang harus mencari data puluhan praktikan dalam satu sesi, akumulasi waktu yang dihemat menjadi signifikan. Terlebih lagi, dibandingkan dengan proses pencarian manual menggunakan spreadsheet yang membutuhkan 10-60 detik per data (berdasarkan hasil wawancara dengan PJ Praktikum), sistem ini memberikan peningkatan efisiensi yang sangat nyata."*

**Kesimpulan:** Gabungkan keduanya dengan porsi seimbang. Jangan terlalu teknis, tapi juga jangan hanya deskriptif. **Selalu hubungkan angka dengan dampak praktis.**

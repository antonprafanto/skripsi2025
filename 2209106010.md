### B2. Arsitektur Deployment Model Random Forest Tidak Jelas

**Masalah:**
Ini adalah **masalah paling kritis**. Proposal menyebut Random Forest diintegrasikan ke ESP32, tapi tidak ada penjelasan teknis sama sekali tentang **bagaimana** model ML bisa berjalan di mikrokontroler dengan RAM hanya ~520KB.

**Kenapa ini penting?**
Inilah yang membuat skripsi ini menjadi skripsi **Informatika**, bukan Teknik Elektro. Tanpa penjelasan ini, proposal hanya terlihat seperti proyek hardware biasa.

**Cara Memperbaiki:**
Tambahkan sub-bab baru di BAB III (misal: 3.5.4 Deployment Model ke ESP32) yang menjelaskan:

1. **Teknik konversi model** -- Library apa yang digunakan untuk mengubah model sklearn menjadi kode C yang bisa di-flash ke ESP32? Contoh library: `emlearn`, `micromlgen`, atau `sklearn-porter`
2. **Estimasi ukuran model** -- Berapa estimasi ukuran model RF dalam bentuk C header (tergantung jumlah tree dan kedalaman)
3. **Proses inferensi di ESP32** -- Bagaimana alur: baca sensor -> masukkan ke model -> dapat prediksi -> aksi (buzzer/MQTT)
4. **Alternatif:** Jika inferensi dilakukan di server/cloud (bukan di ESP32), jelaskan arsitekturnya dan diskusikan trade-off latency

### B7. Metrik Evaluasi Terlalu Sempit

**Masalah:**
Hanya menggunakan confusion matrix (accuracy, precision, recall, F1-score). Untuk sistem **safety-critical** seperti deteksi kebocoran gas, ini kurang memadai.

**Cara Memperbaiki:**
Tambahkan metrik berikut:

| Metrik | Penjelasan | Kenapa Penting |
|--------|------------|----------------|
| **False Alarm Rate (FAR)** | Persentase alarm yang ternyata bukan kebocoran nyata | Ini metrik utama yang ingin direduksi |
| **Missed Detection Rate (MDR)** | Persentase kebocoran nyata yang tidak terdeteksi | Sangat kritis -- kebocoran tidak terdeteksi = bahaya |
| **Response Time** | Waktu dari kebocoran terjadi sampai alarm berbunyi | Relevan untuk keselamatan |
| **ROC-AUC** | Area under ROC curve | Mengukur kemampuan model secara keseluruhan |

### B10. Tidak Ada Penanganan Imbalanced Dataset

**Masalah:**
Dalam kondisi nyata, data "normal" akan **jauh lebih banyak** daripada data "bocor". Ini disebut **imbalanced dataset**, dan Random Forest bisa bias ke kelas mayoritas.

**Cara Memperbaiki:**
Tambahkan strategi penanganan di metodologi:
- **SMOTE** (Synthetic Minority Over-sampling Technique) untuk oversampling kelas minoritas
- **Undersampling** kelas mayoritas
- **Parameter `class_weight='balanced'`** pada scikit-learn RandomForestClassifier
- Atau kombinasi ketiganya

### B13. Tinjauan Pustaka Random Forest Terlalu Dangkal

**Masalah:**
Sub-bab 2.4 (Random Forest) hanya **1 paragraf pendek** tanpa rumus, pseudocode, atau penjelasan mekanisme kerja. Padahal ini adalah **metode utama** skripsi.

**Cara Memperbaiki:**
Perluas menjadi minimal 2-3 halaman yang mencakup:

1. **Konsep Ensemble Learning dan Bagging**
2. **Proses pembangunan pohon keputusan** -- Information Gain, Gini Impurity
3. **Feature Randomization** -- perbedaan RF dengan bagged decision tree biasa
4. **Mekanisme voting** -- majority voting untuk klasifikasi
5. **Rumus Gini Impurity:**
   ```
   Gini(t) = 1 - Sigma p(i|t)^2
   ```
6. **Parameter penting:** n_estimators, max_depth, min_samples_split, max_features
7. **Kelebihan dan kekurangan RF**

### B14. Confusion Matrix Tidak Ada Rumusnya

**Masalah:**
Sub-bab 2.5 menjelaskan confusion matrix secara narasi tapi **tidak menyertakan rumus** precision, recall, F1-score, dan accuracy.

**Cara Memperbaiki:**
Tambahkan rumus:

```
Accuracy  = (TP + TN) / (TP + TN + FP + FN)
Precision = TP / (TP + FP)
Recall    = TP / (TP + FN)
F1-Score  = 2 x (Precision x Recall) / (Precision + Recall)
```

Sertakan juga tabel confusion matrix 2x2:

|  | Prediksi Positif | Prediksi Negatif |
|--|-------------------|------------------|
| **Aktual Positif** | TP (True Positive) | FN (False Negative) |
| **Aktual Negatif** | FP (False Positive) | TN (True Negative) |

### B16. Preheat Sensor MQ-6 Tidak Disebutkan

**Masalah:**
Sensor MQ-6 membutuhkan waktu **pemanasan (preheat) sekitar 24-48 jam** pada penggunaan pertama agar pembacaan stabil. Ini tidak disebutkan di mana pun.

**Cara Memperbaiki:**

### B17. QoS MQTT Tidak Dibahas

**Masalah:**
Untuk sistem keselamatan, level **Quality of Service (QoS)** MQTT sangat penting:
- QoS 0: pesan bisa hilang (tidak aman untuk alarm)
- QoS 1: pesan pasti sampai minimal sekali (disarankan)
- QoS 2: pesan sampai tepat sekali (paling aman, tapi paling lambat)

**Cara Memperbaiki:**
Jelaskan di BAB II (sub-bab MQTT) atau BAB III bahwa QoS 1 atau 2 akan digunakan untuk topik alarm, dengan alasannya.

---

### B18. Keterbatasan Kodular untuk Safety-Critical System

**Masalah:**
Kodular memiliki keterbatasan serius:
- Tidak mendukung background service dengan baik
- Notifikasi bisa tidak muncul jika aplikasi di-kill oleh OS Android
- Tidak ada mekanisme push notification yang reliable

Untuk aplikasi deteksi kebocoran gas, ini bisa **berbahaya** jika notifikasi tidak sampai.

**Cara Memperbaiki:**
Salah satu dari:
- Akui sebagai **batasan penelitian** di BAB I
- Pertimbangkan tambahan mekanisme notifikasi yang lebih reliable (misal: Telegram Bot, atau Firebase Cloud Messaging)
- Jelaskan bahwa buzzer lokal tetap menjadi alarm utama, dan app hanya monitoring tambahan

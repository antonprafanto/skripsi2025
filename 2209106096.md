### 1.1 Latar Belakang ✅ SANGAT BAIK

**Halaman 11-13:**

**Struktur Argumentasi:**
1. ✅ Konteks masalah: Pengelolaan praktikum membutuhkan data teratur
2. ✅ Data kuantitatif: 500-700 praktikan per periode, tersebar di beberapa file spreadsheet
3. ✅ Dampak masalah: Waktu pencarian 2-3 menit per data, potensi duplikasi dan kesalahan
4. ✅ Solusi yang diusulkan: Sistem berbasis web dengan Merge Sort dan Sequential Search
5. ✅ Justifikasi algoritma: Merge Sort O(n log n) stabil, Sequential Search cocok untuk data dinamis

**Kekuatan Luar Biasa:**

✅ **Data Konkret dan Spesifik:**
> "...menyimpan ratusan data mahasiswa dalam satu periode praktikum, yaitu sekitar 500–700 praktikan, ke dalam beberapa file spreadsheet yang terpisah berdasarkan jenis data, seperti file presensi, pretest, posttest, nilai utama periode, dan data induk praktikan."

**Analisis:**
- Data kuantitatif **500-700 praktikan** memberikan gambaran skala masalah
- Fragmentasi file (presensi, pretest, posttest, nilai utama, data induk) menunjukkan kompleksitas administrasi
- Waktu pencarian **2-3 menit per data** menunjukkan inefisiensi yang terukur

✅ **Identifikasi Akar Masalah:**
> "Fragmentasi data praktikan ke dalam berbagai file menyebabkan proses pencarian dan pengurutan data menjadi kurang efisien."

**Analisis:**
- Tidak hanya mengeluhkan gejala, tapi mengidentifikasi **akar masalah**: fragmentasi data
- Dampak jelas: kesulitan penelusuran, duplikasi data, ketidakkonsistenan, kesalahan input

✅ **Gap Analysis yang Kuat:**
> "...spreadsheet menyediakan fitur pengurutan dan pencarian data, mekanisme tersebut tidak terintegrasi lintas file dan tidak berjalan secara otomatis pada seluruh data praktikan."

**Analisis:**
- Menunjukkan bahwa spreadsheet **memiliki fitur**, tapi **tidak mencukupi** untuk skala besar
- Kebutuhan: integrasi lintas file, otomasi, konsistensi

✅ **Justifikasi Algoritma Tepat:**
> "Algoritma Merge Sort mampu mengurutkan data dengan kompleksitas rata-rata O(n log n) dan menghasilkan urutan yang stabil karena prosesnya membagi data menjadi bagian-bagian kecil sebelum digabung kembali (Esau Taiwo et al., 2020)."

**Analisis:**
- Pemilihan Merge Sort atas Quick Sort menunjukkan pemahaman trade-off
- **Merge Sort lebih stabil** dan **kompleksitas konsisten** di semua kondisi (best, average, worst case sama-sama O(n log n))
- Quick Sort memiliki worst case O(n²) jika pivot tidak tepat

> "Sequential Search dapat digunakan untuk melakukan pencarian data secara sederhana tanpa memerlukan struktur data atau kondisi awal tertentu, sehingga sesuai untuk diterapkan pada data yang baru di-input atau belum memiliki struktur tetap."

**Analisis:**
- Justifikasi Sequential Search **tepat**: cocok untuk data dinamis
- Binary Search memerlukan data terurut, Sequential Search tidak
- Relevan untuk sistem praktikum yang datanya terus berubah

**Kelemahan Minor:**

⚠️ **Tidak Ada Baseline Comparison:**
- Latar belakang tidak menyebutkan perbandingan dengan **sorting/searching bawaan database** (MySQL ORDER BY, WHERE, LIKE)
- Dalam praktik web development, sorting di level database umumnya lebih cepat daripada di application layer
- Perlu dijelaskan **kapan dan mengapa** algoritma manual diperlukan

**Rekomendasi Perbaikan:**

```
1. Real-time filtering tanpa database query untuk responsivitas tinggi
```

#### Kelompok 1: Penelitian tentang Merge Sort (Penelitian 1, 3, 6)

**Penelitian 1 (Esau Taiwo et al., 2020):**
> "Penelitian ini melakukan studi komparatif antara dua algoritma Divide and Conquer, yaitu Merge Sort dan Quick Sort... Quick Sort unggul dalam kecepatan, sedangkan Merge Sort memiliki stabilitas lebih tinggi untuk dataset besar dengan kompleksitas waktu O(n log n)."

**Analisis:**
- ✅ **Relevan langsung** dengan pemilihan algoritma Vista
- ✅ Menegaskan trade-off: Quick Sort lebih cepat, Merge Sort lebih stabil
- ✅ Mendukung justifikasi pemilihan Merge Sort untuk sistem dengan kebutuhan stabilitas

**Penelitian 3 (Aditya et al., 2025):**
> "Penelitian menunjukkan bahwa Merge Sort memiliki stabilitas tinggi dengan waktu pemrosesan yang konsisten meskipun ukuran data meningkat... Merge Sort dinilai sesuai diterapkan pada sistem berbasis pendidikan yang memerlukan pengurutan data besar secara real-time."

**Analisis:**
- ✅ **Konteks sama**: sistem pendidikan dengan data mahasiswa
- ✅ Menegaskan **konsistensi performa** Merge Sort
- ✅ Kata kunci "real-time" mendukung implementasi di web application

**Penelitian 6 (Anggreani et al., 2020):**
> "Hasil penelitian menunjukkan bahwa algoritma Merge Sort memiliki waktu eksekusi paling cepat dalam mengurutkan data besar—dengan rata-rata 108,59 ms untuk 3000 data—meskipun membutuhkan memori lebih besar dibanding algoritma lainnya (21,44 MB)."

**Analisis:**
- ✅ **Data kuantitatif sangat berharga**: 108,59 ms untuk 3000 data, memori 21,44 MB
- ✅ Menunjukkan trade-off: kecepatan vs memori
- ✅ Dapat digunakan sebagai **benchmark** untuk evaluasi Vista

**Analisis:**
- ✅ Menunjukkan Sequential Search **efektif untuk skala menengah**

### J.1 LANJUTAN SUB-BAB 3.4 - FLOWCHART ALGORITMA

#### 3.4.9 Flowchart Sequential Search (Gambar III.11) ✅ EXCELLENT

**Halaman 38 (51 dalam PDF):**

**Penjelasan Tekstual:**
> "Flowchart Sequential Search menggambarkan alur proses pencarian data dengan cara menelusuri elemen array secara berurutan dari awal hingga akhir... Sistem menginisialisasi indeks pencarian pada posisi awal array. Sistem kemudian membandingkan nilai elemen pada indeks saat ini dengan data yang dicari. Jika nilai tersebut sama, maka sistem menyimpulkan bahwa data ditemukan dan proses pencarian dihentikan..."

**Analisis Flowchart:**

**Alur:**
1. **Mulai**
2. **Sistem mendeteksi kategori pencarian** (proses)
3. **Input data yang di cari (key)** (input)
4. **Inisialisasi indeks i = 0** (proses)
5. **Decision 1:** Apakah `i < N`? (N = jumlah data)
   - **Tidak** → Data tidak ditemukan → Selesai
   - **Ya** → Lanjut ke Decision 2
6. **Decision 2:** Apakah `array[i] == key`?
   - **Ya** → Data ditemukan → Selesai
   - **Tidak** → `i = i + 1` → Kembali ke Decision 1

**Kekuatan LUAR BIASA:**

1. ✅✅✅ **Logika Algoritma BENAR:**
   - Loop `i < N` memastikan semua elemen diperiksa
   - Perbandingan `array[i] == key` adalah operasi dasar Sequential Search
   - Early exit ketika data ditemukan (efisien)

2. ✅✅✅ **Kategori Pencarian Eksplisit:**
   - "Sistem mendeteksi kategori pencarian" menunjukkan **multiple search criteria**
   - Bisa cari by NIM, Nama, Program Studi, dll
   - Ini **value added** yang tidak umum di penelitian lain

3. ✅✅✅ **Kompleksitas Waktu Tergambar Jelas:**
   - Worst case: Loop sebanyak N kali → **O(n)**
   - Best case: Data di index 0 → **O(1)**
   - Average case: N/2 iterasi → **O(n)**

4. ✅ **Simbol Flowchart Standar:**
   - Oval untuk Start/End
   - Parallelogram untuk Input/Output
   - Rectangle untuk Process
   - Diamond untuk Decision

**Perbandingan dengan Proposal Dinda:**

| Aspek | Vista (Sequential Search) | Dinda (Binary Search) |
|-------|---------------------------|----------------------|
| **Flowchart** | ✅ Ada + Penjelasan detail | ✅ Ada + Penjelasan detail |
| **Kategori Pencarian** | ✅✅ Multi-kategori (NIM/Nama/Prodi) | ⚠️ Tidak disebutkan |
| **Early Exit** | ✅ Ya, ketika data ditemukan | ✅ Ya, ketika data ditemukan |
| **Loop Condition** | `i < N` | `low <= high` (binary) |
| **Kompleksitas** | O(n) | O(log n) |

**Rekomendasi:**


#### 3.4.10 Flowchart Merge Sort (Gambar III.12-III.13) ✅✅✅ EXCEPTIONAL

**Halaman 39-40 (52-53 dalam PDF):**

**Penjelasan Tekstual:**
> "Flowchart Merge Sort menjelaskan alur proses pengurutan data menggunakan metode divide and conquer. Proses dimulai ketika sistem menerima input array data... Sistem memeriksa kondisi jumlah data, jika jumlah elemen kurang dari atau sama dengan satu, maka data dianggap telah terurut... Algoritma Merge Sort dijalankan secara rekursif pada masing-masing subarray hingga setiap subarray hanya memiliki satu elemen. Setelah proses rekursi selesai, sistem melakukan proses penggabungan (merge)..."

**Analisis Flowchart Utama (Gambar III.12):**

**Alur Part 1:**
1. **Mulai**
2. **Input array data** (input)
3. **Menghitung jumlah data (N)** (proses)
4. **Decision:** Apakah `N <= 1`?
   - **Ya** → Langsung ke output (base case rekursi)
   - **Tidak** → Lanjut pembagian
5. **Menentukan titik tengah (mid = N / 2)** (proses)
6. **Flowchart terpotong** → Lanjut ke Gambar III.13

**Analisis Flowchart Lanjutan (Gambar III.13):**

**Alur Part 2:**
7. **Membagi array menjadi subarray kiri dan subarray kanan** (proses dari Part 1)
8. **Proses rekursi:**
   - Panggil Merge Sort pada subarray kiri
   - Panggil Merge Sort pada subarray kanan
9. **Proses Merge:** Bandingkan elemen terkecil dari subarray kiri dan kanan. Masukkan ke array utama
10. **Decision:** Apakah salah satu subarray habis?
    - **Ya** → Masukkan sisa elemen subarray yang belum habis
    - **Tidak** → Loop
11. **Output: Data sudah terurut**
12. **Selesai**

**Kekuatan LUAR BIASA:**

1. ✅✅✅ **REKURSI TERGAMBAR DENGAN JELAS:**
   - Base case `N <= 1` adalah **konsep rekursi fundamental**
   - Flowchart menunjukkan **panggil diri sendiri** pada subarray kiri dan kanan
   - Ini **sangat sulit digambarkan** di flowchart, Vista berhasil!

2. ✅✅✅ **DIVIDE AND CONQUER EKSPLISIT:**
   - **Divide:** Membagi array di titik tengah `mid = N/2`
   - **Conquer:** Rekursi pada subarray kiri dan kanan
   - **Combine:** Proses Merge dengan perbandingan elemen terkecil

3. ✅✅✅ **PROSES MERGE DETAIL:**
   - Bandingkan elemen terkecil dari kedua subarray
   - Masukkan ke array utama secara terurut
   - Tangani sisa elemen subarray yang belum habis
   - Ini adalah **inti dari Merge Sort**

4. ✅✅ **PEMBAGIAN FLOWCHART 2 HALAMAN:**
   - Karena kompleksitas algoritma, Vista bijak membagi menjadi 2 gambar
   - Gambar III.12: Inisialisasi dan divide
   - Gambar III.13: Rekursi dan merge
   - Ini menunjukkan **pemahaman mendalam** bahwa Merge Sort kompleks

**Perbandingan dengan Proposal Dinda (Quick Sort):**

| Aspek | Vista (Merge Sort) | Dinda (Quick Sort) |
|-------|--------------------|--------------------|
| **Flowchart Rekursi** | ✅✅ 2 halaman, sangat detail | ✅ 1 halaman, cukup detail |
| **Divide Strategy** | `mid = N/2` (selalu tengah) | Pivot selection (fleksibel) |
| **Merge Process** | ✅✅ Detailed merge dengan loop | N/A (in-place sorting) |
| **Base Case** | `N <= 1` | `low >= high` |
| **Kompleksitas** | O(n log n) semua kasus | O(n²) worst case |
| **Space Complexity** | O(n) - butuh array bantu | O(log n) - in-place |

**Analisis Mendalam:**

**Kenapa Vista Pilih Merge Sort vs Quick Sort?**

1. **Stabilitas Sorting:**
   - Merge Sort: **Stable** (elemen sama tetap urutan relatif)
   - Quick Sort: **Unstable** (elemen sama bisa tukar posisi)
   - Untuk data praktikan dengan **multiple sort criteria** (nama, NIM, prodi), **stable sort lebih aman**

2. **Predictable Performance:**
   - Merge Sort: **O(n log n)** di semua kasus (best, average, worst)
   - Quick Sort: O(n²) di worst case (data sudah terurut)
   - Untuk sistem akademik yang butuh **reliable performance**, Merge Sort lebih aman

3. **Trade-off:**
   - Merge Sort: Butuh **O(n) extra space** untuk array bantu
   - Quick Sort: **In-place**, hanya O(log n) stack space
   - Vista mengorbankan memory untuk stabilitas dan predictability

**Rekomendasi:**

```
Priority 1 (WAJIB TAMBAHKAN):
✅ Tambahkan penjelasan kenapa pilih Merge Sort vs Quick Sort
✅ Jelaskan trade-off space complexity O(n) untuk stabilitas
✅ Tambahkan pseudocode Merge Sort
✅ Tambahkan contoh step-by-step sorting 8 elemen

Priority 2 (Nice to Have):
✅ Tambahkan tabel perbandingan Merge Sort vs Quick Sort vs Bubble Sort
✅ Jelaskan implementasi array bantu di Laravel/PHP
```
